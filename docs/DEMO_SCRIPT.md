# 产品级演示脚本 (Demo Script)

本文档面向产品、研发及验收方，旨在提供一套标准的端到端（E2E）演示流程，以展示“三栏规则工作台”当前阶段实现的核心业务闭环能力。

## 演示环境准备

### 1. 启动后端
- **操作动作**：在终端执行 `./start_api.sh`
- **预期现象**：自动执行 8 个后端集成测试（全部 PASS），并最终在 `0.0.0.0:8000` 启动 FastAPI 服务器。
- **强调能力点**：
  - 后端提供真实的 FastAPI RESTful 接口（关闭了 Mock）。
  - 已通过 Pytest 构建了针对 DTO 和业务流的回归防线。

### 2. 启动前端
- **操作动作**：在另一个终端执行 `./start_frontend.sh`
- **预期现象**：依赖安装完成后，在 `http://localhost:5173` 启动 Vite 服务器。
- **强调能力点**：
  - 前端具备 `VITE_USE_MOCK_API` 双通道配置，演示时已强制切为真实接口。
  - 前后端分离架构、API Adapter 适配层已全面生效。

---

## 核心工作流演示

### 3. 展示 Baseline Tree (基线树)
- **操作动作**：在浏览器打开前端页面。
- **预期现象**：左栏自动请求 `GET /api/baselines` 并渲染出基线树（包含 `B001` 的各个版本）。
- **强调能力点**：
  - 左栏节点支持“类型 + 关系”建模（Draft / Version / Rollback Candidate）。
  - 后端真实返回的树形结构被 Adapter 正确映射。

### 4. 展示 History Detail (历史版本详情)
- **操作动作**：在左栏点击任意历史版本节点（如 `v1.9` 或 `v1.10`）。
- **预期现象**：
  - 右栏出现 `VersionMeta` 模块，展示发布时间、发布人等元信息。
  - 顶部操作区出现“查看差异 (Compare)”和“回滚 (Rollback)”按钮。
- **强调能力点**：
  - 版本详情数据直接来自 `GET /api/baselines/{id}/versions/{version_id}`。
  - 页面状态机（Reducer）驱动，只读模式下的 UI 完全锁定，防误触。

### 5. 展示 Diff (版本差异)
- **操作动作**：在版本详情页面，点击右上角的“Compare”按钮。
- **预期现象**：
  - 状态机切换到 `diff` 模式。
  - 右栏显示差异统计（Added / Removed / Modified）。
  - 中栏渲染出具体的变更规则前后的对比差异。
- **强调能力点**：
  - Diff 上下文 (`diffContext`) 完全由 Reducer 接管。
  - 后端真实的 `GET /api/baselines/{id}/diff` 接口已打通。

### 6. 展示 Validate 拦截与字段定位
- **操作动作**：
  - 选中左栏的“工作草稿 (Working Draft)”。
  - 在中栏修改一条规则的配置（例如使用不支持的 `operator` 或非法的 `rule_type`）。
  - 右栏面板点击“发布”按钮，触发 `POST /api/rules/draft/validate`。
- **预期现象**：
  - 页面提示校验失败。
  - 右栏展示具体的错误 Issue 列表。
  - **点击右栏的一条错误信息**，中栏的表单对应字段（如 `params`）会高亮（CSS Ring）并滚动到可视区域。
- **强调能力点**：
  - 后端真实的领域模型校验 `RuleCompiler` 拦截了非法配置。
  - 右侧栏驱动左侧表单的物理联动交互，极大降低了复杂表单的排错成本。

### 7. 展示 Publish Blocked (发布阻断)
- **操作动作**：在中栏输入触发后端底线阻断的关键字（例如 `{"block": true}`），再次点击发布。
- **预期现象**：
  - 状态机经历 `publish_checking` -> `publish_blocked`。
  - 发布被阻断，右栏明确展示阻断原因（如“包含敏感关键字”）。
- **强调能力点**：
  - 发布流彻底闭环，`publish_blocked` 是一个稳定且可停留的状态，不会闪退。
  - 用户必须修复问题后才能重新触发发布。

### 8. 展示 Publish Success 与新版本出现
- **操作动作**：修复报错后，点击发布。
- **预期现象**：
  - 发布成功，右栏提示成功。
  - **自动触发导航**：左栏树刷新，最新版本（例如 `v1.12`）出现并被自动高亮选中。
  - 中栏自动变为只读态。
- **强调能力点**：
  - 真实后端完成了原子的 `tempfile` JSON 持久化写入。
  - 状态机通过 `TRIGGER_POST_PUBLISH_NAVIGATION` 动作精确完成了发布后的定位与状态清理。

### 9. 展示 Rollback Candidate 与回填编辑态
- **操作动作**：
  - 选中任意历史版本（如 `v1.8`），点击“Rollback”。
- **预期现象**：
  - 状态机切换到 `rollback_ready_edit` 模式。
  - 左栏生成一个明确的“回滚候选 (Rollback Candidate)”草稿节点。
  - 中栏的标题区清晰提示“此为来源于 v1.8 的历史草稿，尚未发布，仍需复核”。
- **强调能力点**：
  - `POST /api/rules/rollback` 接口成功下发了回填草稿数据。
  - 回滚操作不会直接覆盖现有主干，而是作为 Candidate 显式存在，确保数据安全和二次编辑能力。
