# 连线端口检查工具：项目立项说明 / 架构概览

## 1. 项目背景

在工程交付、设计复核、设备连线审查等场景中，设备、端口、链路、状态、拓扑关系等数据通常来源于 Excel 或其他半结构化表格。  
这类数据存在以下典型问题：

- 表头结构复杂，包含双层表头、合并单元格、分区表头
- 命名不统一，字段质量参差不齐
- 端口与链路关系复杂，人工排查效率低
- 检查标准会持续变化，硬编码规则难维护
- 缺乏规则版本管理、差异追踪和发布治理能力

因此，本项目的目标不是开发一个一次性检查脚本，而是构建一个可持续演进的本地规则检查平台。

---

## 2. 项目目标

项目要实现的是一条完整的工程规则闭环：

**数据导入 → 数据识别 → 标准化 → 规则检查 → 结果输出 → 规则编辑 → 编译校验 → 发布治理 → 版本迭代**

核心目标包括：

1. 将原始工程表格数据转化为统一内部模型  
2. 通过规则引擎自动发现设备、端口、链路、拓扑相关问题  
3. 将规则从代码中外置，形成可治理的规则资产  
4. 支持规则编辑、草稿预编译、字段级错误回挂  
5. 支持规则发布、版本管理、差异分析和回滚  
6. 为后续 GUI 与 AI 自动生成能力提供稳定底座  

---

## 3. 项目定位

本项目本质上是：

**一个面向工程场景的本地化连线/端口检查与规则治理平台。**

它不是只做“校验某几列字段”的脚本，而是一个围绕“规则资产”构建的工程质检系统。

平台长期能力方向包括：

- 工程数据识别
- 结构化规则检查
- 规则目录化
- 规则编辑器
- 编译与验证桥接
- 发布与治理
- 版本差异与回滚
- AI 辅助规则生成

---

## 4. 核心功能范围

### 4.1 数据导入与识别
负责解析 Excel 或类似工程表格，识别复杂表头与分区信息，并抽取统一事实模型。

典型能力：
- 工作表识别
- 双层表头识别
- 合并单元格处理
- 起始端 / 目的端分区识别
- 设备名称、端口、状态等字段提取
- 原始行映射为内部对象

### 4.2 数据标准化与契约校验
负责在规则执行前进行基础合法性检查。

典型能力：
- 必填校验
- 类型校验
- 枚举校验
- 格式校验
- 行内约束校验
- 字段归一化

### 4.3 规则引擎检查
负责执行业务层面的语义检查。

主要规则类型：
- `single_fact`
- `group_consistency`
- `threshold`
- `topology`

典型场景：
- 非法状态
- 命名违规
- 双端不一致
- 重复占用
- 阈值超限
- 非法连接
- 拓扑断裂

### 4.4 结果输出与问题展示
负责生成结构化检查结果，支持概览、聚合、明细、证据、差异等层次。

### 4.5 规则目录与消费层
通过 Rule Catalog 统一描述系统支持的规则类型、参数 schema、默认示例、表单定义和能力元信息。

### 4.6 规则编辑器
基于 Catalog 驱动规则表单渲染，实现规则类型选择、默认值填充、表单校验和草稿生成。

### 4.7 编辑器到治理链路桥接
将前端 `RuleDraft` 转换到底层规则定义，执行编译与结构验证，并将错误统一映射为 UI 可理解的字段级错误。

### 4.8 规则发布与版本治理
将通过校验的规则草稿发布为正式 baseline，并纳入版本管理体系。

### 4.9 差异分析与回滚
支持规则版本 diff、历史追踪和回滚，降低规则升级风险。

### 4.10 AI 辅助规则生成
在规则治理体系成熟后，引入自然语言生成规则草稿、参数补全和候选摘要生成能力。

## 5. 系统架构思路

项目遵循分层设计，避免 UI、应用编排、领域模型、基础设施和横切能力耦合在一起。

推荐的整体分层如下：

### 5.1 Presentation 层
负责桌面 UI、交互视图、用户操作入口。

职责：
- 展示规则列表、编辑表单、问题结果、发布摘要
- 调用 application service
- 不直接依赖底层 compiler / repository / domain internals

### 5.2 Application 层
负责业务编排，是系统主流程的组织者。

职责：
- 编排导入、检查、规则编辑、编译预览、发布流程
- 输出面向 UI 的 DTO / ViewModel
- 协调 Domain 与 Infrastructure
- 不承载底层框架细节

典型服务：
- CheckRunService
- RuleEditorMVPService
- RuleEditorGovernanceBridgeService
- RulePublishWorkflowService（下一步重点）

### 5.3 Domain 层
负责稳定的核心业务模型和规则语义。

职责：
- 定义设备、端口、链路、拓扑等核心模型
- 定义规则协议、CompiledRule、RuleMeta、Capability 等领域概念
- 定义领域约束与不可变语义

### 5.4 Infrastructure 层
负责持久化、文件读写、外部适配和底层实现。

职责：
- Excel 解析
- baseline 存储
- repository 实现
- 文件系统适配
- 编译器外部装配与加载

### 5.5 Cross-cutting 层
负责日志、配置、异常、公共工具、序列化等横切能力。

## 7. 当前阶段判断

当前项目已经不再是“概念设计期”，也不是“只有底层协议没有产品入口”的状态。  
它已经完成了规则平台最关键的一段骨架建设：

**Catalog → Editor → Draft → Compile Preview → Validate → Candidate Summary**

这说明系统已经具备从规则定义走向规则治理的基础能力。

当前最关键的缺口不是继续增加概念层，而是补齐真正的“发布闭环”，也就是：

- 发布服务
- baseline 落盘
- 版本号生成
- 发布摘要
- 发布历史

只有这一段完成，系统才真正从“可编辑”进入“可治理”。

---

## 8. 当前最关键的下一步

当前优先级最高的单一任务是：

### Rule Publish Workflow MVP

目标：
- 接收 `RuleDraftView`
- 强制执行 compile + validate
- 拒绝带错误的草稿发布
- 生成新的 baseline version
- 写入治理存储
- 返回结构化发布结果

该阶段完成后，系统将形成最小治理闭环：

**编辑 → 草稿 → 预编译 → 校验 → 发布 → 基线生效**

## 10. 其他模型接手时应先理解的结论

任何后续接手本仓库的模型，在开始开发前，应先明确以下事实：

1. 项目主线不是“写几个规则函数”，而是构建规则平台  
2. Rule Catalog 是规则编辑器与治理层的重要中心  
3. RuleEditorMVPService 负责 UI 侧规则创建闭环  
4. RuleEditorGovernanceBridgeService 负责 Editor 与底层 Compiler/Validate 的桥接  
5. 当前最缺的不是继续做更多编辑能力，而是补齐发布治理闭环  
6. 后续 GUI、Diff、回滚、AI 生成，都应建立在发布链路稳定的前提下  
7. 必须遵守既有分层架构，避免 UI 直接下沉到底层编译或存储逻辑
