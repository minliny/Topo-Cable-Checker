# 下一阶段迭代路线图 (Next Phase Roadmap)

本文档基于当前已完成的“三栏规则工作台 MVP”版本，规划了后续（Phase 2/3）的演进路线。路线图分为三个优先级：P1（真实用户可感知增强）、P2（工程稳健性增强）、P3（长期产品化能力）。

---

## P1：真实用户可感知增强

### 1. 引入 Monaco Editor 提升编辑体验
- **目标**：将中栏原生的 JSON 文本域替换为 `@monaco-editor/react`，支持 JSON Schema 校验、代码折叠与语法高亮。
- **价值**：极大提升规则编写人员的效率，且能将 `Validate` 接口返回的 `field_path` 直接映射为代码编辑器的行号和红色波浪线（Markers），实现精准报错。
- **依赖**：前端集成 Monaco Editor 库及相关 Web Worker。
- **是否阻断当前验收**：**否**。当前基于原生 DOM 的高亮（CSS Ring）已能满足基础定位需求。

### 2. 深度 JSON Diff 对比视图
- **目标**：在 `GET /api/baselines/{id}/diff` 接口中引入 `deepdiff` 或 `dictdiffer`，并配合前端使用类似 `react-diff-viewer` 的组件。
- **价值**：在 Compare 视图中，不仅能看出“哪条规则变了”，还能以红绿背景精确看出“规则内部的哪个字段变了”。
- **依赖**：后端需替换浅层 Diff 算法，前端需引入 Diff 专用组件。
- **是否阻断当前验收**：**否**。当前已能正确计算 Added/Removed/Modified 的规则级变更。

---

## P2：工程稳健性增强

### 1. 持久化层迁移至关系型数据库 (PostgreSQL)
- **目标**：废弃当前的 `baselines.json` 临时文件写入方案，将 `BaselineRepository` 迁移至 SQLAlchemy ORM 模型。
- **价值**：支持海量基线数据存储、复杂的树状关系查询优化、严格的事务隔离机制。
- **依赖**：部署 PostgreSQL 实例，完成数据库 Schema 设计与 Alembic 迁移脚本。
- **是否阻断当前验收**：**否**。当前基于 `tempfile` 的原子写入方案已能保证单机无损演示。

### 2. 发布并发冲突检测 (Optimistic Locking)
- **目标**：在 `publish` 接口增加对 `parent_version` 的校验，若用户提交的 `parent_version` 落后于数据库中的最新版本，则阻断发布并提示“基线已被他人修改”。
- **价值**：防止多用户并发编辑时发生数据互相覆盖的灾难性后果。
- **依赖**：后端在持久化层增加版本号比对逻辑。
- **是否阻断当前验收**：**否**。当前版本主要用于验证状态机闭环与单用户交互，不涉及高并发。

---

## P3：长期产品化能力

### 1. 接入企业级 RBAC 权限管控
- **目标**：在 FastAPI 中接入 `Depends(get_current_user)` 提取真实的 JWT Token，并在前端根据用户角色控制 UI 操作（如：普通只读用户无法看到“发布”和“回滚”按钮）。
- **价值**：实现细粒度权限控制，`version_history_meta` 中的 `publisher` 字段替换为真实用户名，满足企业安全审计合规要求。
- **依赖**：企业统一认证中心 (SSO) / OAuth2 的对接。
- **是否阻断当前验收**：**否**。当前通过硬编码 `"admin"` 已能跑通元信息流转。

### 2. 发布审批工作流 (Approval Workflow)
- **目标**：在前端状态机（Reducer）中引入 `pending_approval` 状态，后端新增审批工单接口。规则发布需经主管审批后才正式落库为新版本。
- **价值**：贴合金融、政务等对风控规则发布极度敏感行业的业务诉求。
- **依赖**：引入轻量级工作流引擎（如基于状态机的自研流或 Camunda 等），并增加审批看板页面。
- **是否阻断当前验收**：**否**。当前的一键发布流已能自证“编辑->校验->阻断/成功->新版本”的核心闭环。
