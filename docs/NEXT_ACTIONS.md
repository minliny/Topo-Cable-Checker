# 下一阶段执行清单 (Next Actions)

本文档基于《项目定位决策 (Project Positioning)》中选定的**产品化路径 (Rule Governance Tool)** 目标，并结合了从真实用户视角（非研发人员）进行端到端演示后的反馈，输出了重排后的具体任务清单。

## 任务重排与用户痛点映射

我们决定将本系统深度发展为**企业级产品（权限、审批、审计）**，以下是基于真实用户反馈重排后的优先级清单。**所有任务均强绑定了演示中发现的“用户问题”**，杜绝无目标的工程自嗨。

### P0：必须修（核心使用阻塞点）

| 序号 | 任务名 | 绑定的用户问题 | 描述 | 状态 |
| :--- | :--- | :--- | :--- | :--- |
| **1** | **集成 Monaco Editor** | **JSON 手填门槛极高**：纯文本框中输入 JSON 极易遗漏引号或括号，导致低级语法错误，用户挫败感强。 | 替换当前的 `<TextArea>` 为 `@monaco-editor/react`，实现 JSON 语法高亮、自动补全与防呆提示，彻底解决输入门槛。 | Todo |
| **2** | **重组核心行动按钮区** | **操作视线跳跃**：在右栏看完校验错误后，需要回到中栏顶部寻找“发布/重新校验”按钮，打破了操作直觉。 | 优化核心操作流，在右栏增加 Publish 按钮的镜像或快捷入口，保持用户视线聚焦。 | Todo |
| **3** | **补齐 Version Meta 渲染** | **历史元数据缺失**：历史版本右侧栏未渲染真实的发布人和时间，导致“审计与追溯”这一企业级核心价值感知变弱。 | 将真实下发的 `version_history_meta` 信息渲染到右侧栏的 Alert 下方，直观展示“发布人、时间、摘要”。 | Todo |

### P1：体验提升（核心业务价值放大）

| 序号 | 任务名 | 绑定的用户问题 | 描述 | 状态 |
| :--- | :--- | :--- | :--- | :--- |
| **4** | **集成深度 JSON Diff 算法** | **Diff 视图技术感过重**：深层 JSON 的直接对比让非技术人员难以理解具体的业务变更（如“阈值从 5 变为 10”）。 | 引入深度且对业务友好的 Diff 算法展示（如 `deepdiff` 和 `react-diff-viewer`），精准高亮到具体的 Key-Value 变化。 | Todo |
| **5** | **统一 Publish Blocked 展示** | **拦截信息展示位置不统一**：当前底线阻断在中栏展示，而常规校验在右栏展示，用户不知道该看哪边。 | 将所有错误和拦截信息统一到右栏展示，降低认知负担。 | Todo |

### P2：工程能力（企业级底座演进）

| 序号 | 任务名 | 绑定的用户问题 | 描述 | 状态 |
| :--- | :--- | :--- | :--- | :--- |
| **6** | **乐观锁并发防覆盖** | **多人协作的潜在覆写风险**：在真实业务场景中，多名风控人员可能同时编辑同一基线，导致脏写。 | 在 Publish API 中增加 `parent_version` 参数校验，若提交版本落后于 DB 最新版本则阻断。 | Todo |
| **7** | **接入真实用户身份 (RBAC)** | **实名操作留痕缺失**：企业产品需要明确知道是谁做出了高危变更，不能只用 admin 兜底。 | 在 FastAPI 接入 JWT 鉴权，将版本记录与真实员工域账号绑定，控制敏感操作权限。 | Todo |
| **8** | **关系型数据库迁移 (PostgreSQL)** | **文件存储的并发与审计查询瓶颈**：随着版本历史增多和多人并发，JSON 文件原子写入无法支撑复杂的历史查询。 | 废弃 `baselines.json`，将持久化层迁移至 SQLAlchemy ORM，支撑企业级数据底座。 | Todo |
| **9** | **引入发布审批流** | **政企客户合规审计要求**：演示中用户提出“希望能看到谁审批了这个变更”。 | 在状态机中新增 `pending_approval` 状态，高危规则变更需主管审批后方可生效。 | Todo |

> *注：上述任务的执行将支撑我们在接下来的 1 个月内，从一个具备复杂状态机的“高质量 Demo”正式跃升为可交付给业务团队试用的“专业规则治理工作台”。*
